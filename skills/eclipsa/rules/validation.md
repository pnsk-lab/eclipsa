# Validation

## Normal App Commands

Use the commands generated by a standard Eclipsa app:

- `bun run dev`
- `bun run build`
- `bun run start`
- `bun run typecheck`

If the app uses `vite-plus`, also prefer the matching `vp` commands that the project already defines.

## What To Check

- The app renders correctly on first load.
- Navigation through `Link` and `useNavigate()` behaves correctly.
- Loader-backed content appears on first SSR and after client navigation.
- Action state updates correctly for pending, success, and error paths.
- Metadata updates on route navigation when relevant.
- Browser-only code does not break SSR.

## Quick Validation Matrix

- Route or layout work: check first SSR render, then click through `Link` navigation.
- `useNavigate()` work: trigger the actual event handler, not just direct URL entry.
- Loader work: check initial data, loading state, error state, and manual `load()`.
- Action work: check pending, success, error, and `lastSubmission` if the UI uses it.
- Metadata work: check title and meta tags on first render and after navigation.
- Resume-sensitive work: reload the page and confirm interactive state resumes without replacing the whole document.

## When Features Are Involved

- For loader work, verify both initial route load and manual `load()` calls.
- For action work, verify both form submission and programmatic `action()` calls when both are used.
- For layout work, verify shared state survives client-side route changes.
- For route params, verify dynamic and catch-all routes with realistic URLs.
